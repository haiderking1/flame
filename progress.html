What was built

Per-turn self-improvement loop landed

  • NudgeTracker (nudge-tracker.ts) — memory gate per user turn, skill gate per tool-issuing assistant turn; resets on fire and whenever skill_manage succeeds. 0 disables a gate.
  • runBackgroundReview (background-review.ts) — forks a headless Agent after qualifying turns, off the user-facing path. Reuses the parent streamFn/model/sessionId/base system prompt verbatim (provider prefix-cache parity), restricts tools to a memory + skills whitelist, shares the parent MemoryStore so writes hit disk, caps iterations, dedupes the action summary, best-effort.
  • Review prompts (review-prompts.ts) — the memory / skill / combined prompts ported near-verbatim from background_review.py, retargeted to flame tool names.
  • AgentSession wiring — per-turn cadence on turn_end, memory gate at prompt start, fire-and-forget review after the run drains, emits a self_improvement_review event now rendered in the TUI via showStatus.

Inactivity curator — ops & CLI landed

  • curator.ts — deterministic age-based transitions (active → stale → archived) from each skill dir's most recent file mtime; pinned skills bypass; archive-never-delete (moves to .archive/); optional LLM consolidation fork using a ported curator prompt. Now supports dry-run (simulate, zero writes) and a force path for manual runs.
  • Per-run reports — every pass writes REPORT.md (markdown transition table, pinned list, consolidation results, dry-run banner) and run.json under <skills>/.curator_runs/<id>/.
  • /curator command suite — status, run (--dry-run), pin/unpin, pause/resume, restore [id]. Registered in the slash-command palette and surfaced through the interactive mode.
  • curator-state.ts — .curator_state (lastRunAt, pause, pins, per-skill lifecycle), atomic writes.
  • curator-backup.ts — pre-run recursive-copy snapshots + manifest, pruned to keep N, plus restoreSkillsSnapshot that takes a rollback-safety snapshot before replacing the tree (rollback is itself undoable).
  • Session-start trigger — interval-gated; emits a curator_run event (also rendered). Opt-in via settings.

Skill-tool TUI summary lines landed · new

  • Added renderCall/renderResult to skill_manage, skill_view and skills_list (matching the memory tool). Creating a skill now shows skill_manage create <name> → <path> instead of a raw JSON blob — the gap that made flame feel quieter than hermes on skill writes. Errors, partial, and read paths handled.

Skill usage DB landed · new

  • skill-usage.ts — faithful port of hermes' tools/skill_usage.py. A .usage.json sidecar (snake_case keys, wire-compatible with hermes' own file) tracks per-skill use/view/patch counts + timestamps, created_at, lifecycle state, pinned, and provenance (created_by).
  • Bumps wired exactly like hermes — skill_view bumps both view and use (loading a skill to act on it is a use; the stale timer keys off last_used_at); skill_manage marks created_by: agent only in the background-review/curator context (foreground user creates stay off-limits), bumps patch on patch/edit, and forgets on delete.
  • Curator rewired — transitions now anchor on each skill's real last_activity_at (→ created_at) instead of file mtime, and only touch agent-created skills. Per-skill state + pins moved out of .curator_state into the usage record, exactly as hermes keeps them.

Remaining mechanism differences by design

Two small hermes refinements are all that's left. They are implementation choices, not absent features — the user-visible workflow is equivalent.

A. Consolidation / pruning reconciliation — roadmap #5

The LLM consolidation pass runs and skill_manage delete already takes absorbed_into. What's not ported is hermes' post-hoc _classify_removed_skills that parses the run's YAML to label each removal consolidated vs pruned for reference migration — a reporting refinement.

B. Idle-since-activity trigger — roadmap #8

The curator is interval-gated at session start. Hermes additionally requires a minimum idle window since the last activity. Adding that gate would prevent any chance of competing with an active turn.

Explicitly out of scope (by earlier project decisions)

  • No bundled skills (agent grows its own library — the stated source of value).
  • External memory providers (honcho / mem0 / supermemory), <memory-context> scrubber, session_search.
  • codex_app_server runtime downgrade for the review fork (flame has no codex runtime).
